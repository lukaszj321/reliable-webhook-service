## PostgreSQL

PostgreSQL is the project's primary database. It currently stores webhook endpoint configurations,
webhook event records, and completed webhook delivery attempt records. The local PostgreSQL
instance runs through Docker Compose. See [Development setup](development.md) for environment
setup, service startup, and local validation instructions.

## Connection configuration

`DATABASE_URL` provides the connection string used by the application, Alembic, and tests.
SQLAlchemy connects to PostgreSQL through the Psycopg driver. The URL format is:

```text
postgresql+psycopg://username:password@host:port/database
```

The public local development values in `.env.example` are:

```dotenv
DATABASE_URL=postgresql+psycopg://reliable_webhook:reliable_webhook@127.0.0.1:5432/reliable_webhook
POSTGRES_PORT=5432
POSTGRES_USER=reliable_webhook
POSTGRES_PASSWORD=reliable_webhook
POSTGRES_DB=reliable_webhook
```

If the host port changes, `POSTGRES_PORT` and the port in `DATABASE_URL` must remain consistent.
The local `.env` file is ignored by Git and should not contain credentials intended for production.

## Alembic migrations

Apply all missing migrations:

```powershell
python -m alembic upgrade head
```

Downgrade the database to the state before the first migration:

```powershell
python -m alembic downgrade base
```

> **Warning:** Downgrading to `base` removes all application tables currently managed by Alembic.
> Use this command only intentionally in a local development environment.

Check whether the ORM metadata contains schema changes without a corresponding migration:

```powershell
python -m alembic check
```

## Database schema

### webhook_endpoints

The `webhook_endpoints` table stores configurations for webhook destination addresses.

| Column | PostgreSQL type | Nullable | Default | Description |
|---|---|---:|---|---|
| `id` | `UUID` | No | Generated by the application | Primary key |
| `name` | `VARCHAR(255)` | No | None | Human-readable endpoint name |
| `target_url` | `VARCHAR(2048)` | No | None | Destination URL |
| `is_active` | `BOOLEAN` | No | `true` | Whether delivery to the endpoint is enabled |
| `created_at` | `TIMESTAMP WITH TIME ZONE` | No | `now()` | Creation timestamp |
| `updated_at` | `TIMESTAMP WITH TIME ZONE` | No | `now()` | Last modification timestamp |

### webhook_events

The `webhook_events` table stores events associated with an existing webhook endpoint
configuration. Persistence of these records does not provide webhook delivery.

| Column | PostgreSQL type | Nullable | Default | Description |
|---|---|---:|---|---|
| `id` | `UUID` | No | Generated by the application | Primary key |
| `endpoint_id` | `UUID` | No | None | Foreign key to `webhook_endpoints.id` |
| `event_type` | `VARCHAR(255)` | No | None | Event type identifier |
| `payload` | `JSONB` | No | None | Structured webhook event payload |
| `created_at` | `TIMESTAMP WITH TIME ZONE` | No | `now()` | Creation timestamp |

- `endpoint_id` has the non-unique index `ix_webhook_events_endpoint_id` and references
  `webhook_endpoints.id`.
- The foreign key does not use `ON DELETE CASCADE`. An endpoint with associated events cannot be
  deleted until those events are removed, while deleting events does not delete their endpoint.
- Event UUIDs are generated by the application. PostgreSQL assigns `created_at` through the server
  default `now()`.
- `payload` uses PostgreSQL `JSONB` and supports nested objects, lists, numbers, Boolean values, and
  `null`. The model requires the top-level payload to be a JSON object.
- In-place mutations of nested JSONB values are not tracked through `MutableDict`, because
  `MutableDict` is not configured.
- `WebhookEvent` has no ORM relationship with `WebhookEndpoint`. The association is represented by
  `endpoint_id` and the database foreign key constraint.

### webhook_delivery_attempts

The `webhook_delivery_attempts` table stores completed attempts associated with an existing
`WebhookEvent`.

One event can have multiple attempts, and each attempt belongs to exactly one event. An attempt
represents a completed delivery attempt; the table does not represent `pending`, `processing`, or
`retrying` state. Records can be persisted through the ORM but are not currently created
automatically.

| Column | PostgreSQL type | Nullable | Default | Description |
|---|---|---:|---|---|
| `id` | `UUID` | No | Generated by the application | Primary key |
| `event_id` | `UUID` | No | None | Foreign key to `webhook_events.id` |
| `attempt_number` | `INTEGER` | No | None | Positive sequential attempt number for one event |
| `outcome` | `VARCHAR(32)` | No | None | Completed outcome: `succeeded` or `failed` |
| `target_url` | `VARCHAR(2048)` | No | None | Snapshot of the URL used for this attempt |
| `response_status_code` | `INTEGER` | Yes | None | HTTP status code when a response was received |
| `error_message` | `TEXT` | Yes | None | Concise delivery failure information |
| `duration_ms` | `INTEGER` | No | None | Attempt duration in milliseconds |
| `attempted_at` | `TIMESTAMP WITH TIME ZONE` | No | `now()` | Attempt timestamp |

#### Constraints and indexes

- `id` is the primary key.
- `event_id` is a foreign key to `webhook_events.id` and does not use `ON DELETE CASCADE`.
- `ix_webhook_delivery_attempts_event_id` is a non-unique index on `event_id`.
- `uq_webhook_delivery_attempts_event_id_attempt_number` requires each `(event_id, attempt_number)`
  pair to be unique.
- `ck_webhook_delivery_attempts_attempt_number_positive` enforces `attempt_number > 0`.
- `ck_webhook_delivery_attempts_outcome` allows only `succeeded` and `failed`.
- `ck_webhook_delivery_attempts_response_status_code` requires `response_status_code` to be `NULL`
  or between 100 and 599.
- `ck_webhook_delivery_attempts_duration_ms_non_negative` enforces `duration_ms >= 0`.

PostgreSQL does not enforce cross-field dependencies among `outcome`, `response_status_code`, and
`error_message`. The schema technically permits a `succeeded` attempt with a null status code, a
`failed` attempt with any valid status from 100 through 599, and an `error_message` independently
of `outcome`. This is an intentional boundary of the persistence model.

#### Representations of completed attempts

**Successful HTTP attempt**

- `outcome=succeeded`
- `response_status_code` contains the received status, for example `200`
- `error_message` is normally `NULL`

**Failed HTTP response**

- `outcome=failed`
- `response_status_code` contains the received status, for example `503`
- `error_message` can describe the response failure

**Network or transport failure**

- `outcome=failed`
- `response_status_code=NULL` because no HTTP response was received
- `error_message` can contain a concise timeout, DNS, TLS, or connection failure description

Attempt records do not store response bodies, request headers, response headers, stack traces,
structured exception JSON, credentials, or signing secrets.

#### Target URL snapshot

`target_url` is stored on each attempt as a snapshot of the URL used for that specific attempt.
A future change to `WebhookEndpoint.target_url` should not change the historical value stored in an
existing attempt record. The current application does not yet perform delivery or automatically
create these records.

#### Deletion behavior

The dependency chain is:

```text
WebhookEndpoint
â†’ WebhookEvent
â†’ WebhookDeliveryAttempt
```

An endpoint with events cannot be deleted, and an event with delivery attempts cannot be deleted.
Deleting the attempts first allows their event to be deleted; deleting the events then allows their
endpoint to be deleted. No foreign key uses cascade delete, so deleting a parent never
automatically deletes its children.

## ORM behavior

**WebhookEndpoint**

- The application-side SQLAlchemy default generates a UUID when a new ORM object is persisted.
- PostgreSQL assigns `created_at` through the server default `now()`.
- PostgreSQL assigns the initial `updated_at` value through the server default `now()`.
- SQLAlchemy sets `updated_at` during updates performed through the ORM by using `onupdate`.
- After inserting a webhook endpoint, the API commits the session and refreshes the ORM object.

`updated_at` is not maintained by a PostgreSQL trigger. Its update behavior applies to updates made
through SQLAlchemy's ORM configuration.

**WebhookEvent**

- The application-side SQLAlchemy default uses `uuid.uuid4` to generate the event UUID.
- PostgreSQL assigns `created_at` through the server default `now()`.
- The payload is stored as PostgreSQL `JSONB`.
- The model has no `updated_at` column and no ORM relationship with `WebhookEndpoint`.
- The foreign key does not provide cascade delete.
- `POST /webhook-events` verifies the referenced endpoint and persists a `WebhookEvent` object.
- An inactive endpoint can still accept an event through the creation API.
- Alembic manages the `webhook_events` schema. The event creation API uses the existing schema, so
  it does not require a separate migration.

**WebhookDeliveryAttempt**

- The application-side SQLAlchemy default uses `uuid.uuid4` to generate the attempt UUID.
- PostgreSQL assigns `attempted_at` through the server default `now()`.
- The model has no `updated_at` column and no ORM relationship.
- Its foreign key does not provide cascade delete.
- `target_url` stores a snapshot for the individual attempt.
- `response_status_code` and `error_message` are nullable.
- The table stores only completed attempts.
- The current API does not create attempt records automatically.
- Read-only `GET /webhook-events/{event_id}/delivery-attempts` lists stored completed attempts for
  one existing event without creating or modifying them. A top-level
  `GET /webhook-delivery-attempts` endpoint does not exist.
- Synchronous execution of one delivery exists as an application service. It is not triggered
  automatically after event creation, and no public HTTP endpoint starts delivery. Workers,
  background processing, retry, backoff, and replay are not implemented.

**Shared persistence configuration**

- `SessionFactory` uses `expire_on_commit=False`.
- Alembic manages the schema; the project does not use `Base.metadata.create_all()`.

## Validation boundary

The ORM model stores `target_url` as a string and does not validate URL syntax. API requests are
validated separately by Pydantic and accept only HTTP or HTTPS URLs with a maximum length of 2048
characters. Code that uses the ORM model directly bypasses this API validation. See
[Request validation](api/webhook-endpoints.md#request-validation) for the public API rules.

`WebhookEventCreate` and `WebhookEventResponse` define the public event request and response.
`POST /webhook-events` requires `endpoint_id` to be the UUID of an existing endpoint, trims
`event_type` to between 1 and 255 characters, and requires `payload` to be a top-level JSON object.
The payload is stored as PostgreSQL `JSONB`. Nested objects, lists, JSON scalars, and `null` are
supported within the object, but no payload size limit or event-specific schema is configured.
Inactive endpoints can accept events. The API does not deliver events or automatically create
delivery attempts and does not provide retry or replay behavior. `WebhookDeliveryAttempt`
constraints are enforced by PostgreSQL for ORM persistence, but no public request schema or API
endpoint currently accepts delivery attempt data. See
[Webhook event API](api/webhook-events.md) for the complete request and response behavior.
